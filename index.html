关键特性

更快把规则写出来，更早完成验证，也更稳地跑到线上。

01 规则语言

WFL 面向实时关联检测

用专用 DSL 表达阈值判断、时序链、缺失检测、快照关联和风险评分，而不是让规则散落在通用 SQL 或脚本里。

价值 让规则语义稳定、可解释，也便于评审。

02 运行一致性

实时接入与离线回放统一

`daemon` 处理在线流量，`batch` 用于文件回放和复盘，让线上规则和离线验证共用同一套执行语义。

价值 规则上线前能先回放验证，而不是靠生产试错。

03 工具链

工具链闭环

`wfusion config`、`wfl`、`wfgen` 组成配置诊断、规则开发、数据生成和契约验证的完整链路。

价值 把配置排障、规则验证和样本生成放进同一工作流。

04 执行模型

窗口中心的运行时

规则不是直接消费消息，而是围绕窗口 读取增量数据，watermark、lateness 和 eviction 都真正参与执行。

价值 时间语义、缓存和驱逐策略不是外挂，而是执行核心。

05 逐条富化

支持逐条富化

除了窗口规则，还支持 `on each` 做单条事件整理和打分，再由下游规则继续聚合和关联。

价值 适合把逐条语义整理和下游聚合拆层处理。

06 部署形态

轻量单机部署

不依赖 Kafka、外部数据库或重型控制面，适合规则研发、实验环境、单机场景和早期生产部署。

价值 先把关联检测跑起来，再决定是否扩大部署面。

WFL 核心表达能力

WFL 不是通用查询语言，而是把实时检测规则拆成几个稳定、清晰、可组合的核心能力。

events

事件绑定

作用 把输入 window 和过滤条件绑定成规则里的事件别名。

特点 先把“看什么事件”定义清楚，再进入后续匹配和判断。

Event Binding events 
events {
    fail : auth_events && action == "failed"
    scan : fw_events && dport > 0
}
match<key:duration>

时序匹配

作用 描述按实体、按时间窗口的关联检测。

特点 适合表达“先发生 A，再发生 B”或“5 分钟内累计达到阈值”。

Temporal Match match 
match<sip:5m> {
    on event {
        fail | count >= 3;
        scan | count >= 1;
    }
}
on close

缺失与关闭语义

作用 写 timeout、flush、eos 阶段的判断。

特点 不只检测“发生了什么”，也能检测“本该发生但没有发生什么”。

Close Condition on close 
match<query_id:30s> {
    on event {
        req | count >= 1;
    }
    on close {
        resp && close_reason == "timeout" | count == 0;
    }
}
join + entity

关联与归属

作用 关联外部窗口，并明确告警最终属于谁。

特点 把实时事件、外部上下文和归属实体放进同一条规则链。

Join And Entity join 
join geo_lookup snapshot on sip == geo_lookup.ip

entity(ip, fail.sip)
score + yield

输出结果

作用 给规则评分，并产出中间结果或最终告警。

特点 规则不只负责命中判断，还能直接定义输出结构和下游消费方式。

Score And Yield yield 
-> score(70.0)

yield security_alerts (
    sip = fail.sip,
    fail_count = count(fail),
    risk_score = @score
)

运行时模型

WarpFusion 的运行时不是“消息直推规则”，而是以 Window 为中心组织事件时间、缓存与规则执行。

2
运行模式
`daemon` 实时接入，`batch` 离线回放
3
文件格式
`ndjson` / `arrow_framed` / `arrow_ipc`
1
核心链路
Receiver → Router → Window → Engine → Sink
N
规则任务
每条 rule 独立 task，按 cursor 增量读取
Window-Centric

执行闭环

Receiver Router Window Rule Engine Alert Sink

规则任务通过 cursor 从 Window 增量读取数据，并用 `Notify` 驱动执行，确保窗口语义直接影响规则结果。

事件时间控制

watermark / lateness / eviction

  • 支持基于事件时间的 watermark
  • 支持 `allowed_lateness` 和迟到丢弃策略
  • 支持按时间和内存做窗口驱逐
  • 关闭顺序按生产者先停、消费者后停设计，避免丢告警

三文件模型

逻辑定义、检测规则和运行参数分层，是 WarpFusion 最重要的产品边界之一。

`.wfs` Window Schema

定义 stream、time、over 与字段 schema，是规则和运行时共享的逻辑数据契约。

`.wfl` Detection Rule

定义 `events`、`match`、`join`、`entity`、`yield`，把检测语义固定在可解释的 DSL 里。

`fusion.toml` Runtime Config

定义 mode、sources、runtime、window defaults、vars 和 sinks，把部署参数和业务规则解耦。

Window Schema security.wfs 
window auth_events {
    stream = "syslog"
    time = event_time
    over = 5m

    fields {
        sip: ip
        username: chars
        action: chars
        event_time: time
    }
}
Runtime Config fusion.toml 
mode = "daemon"
sinks = "sinks"

[[sources]]
type = "tcp"
name = "ingress_tcp"
listen = "tcp://127.0.0.1:9800"

[runtime]
executor_parallelism = 2
rule_exec_timeout = "30s"
schemas = "schemas/*.wfs"
rules   = "rules/*.wfl"

适合什么场景

WarpFusion 更适合实时关联与风险分析，不追求把所有问题都收敛成一个通用流平台。

1

多步攻击链检测

例如 5 分钟内同一源 IP 连续登录失败 3 次后又发起扫描或横向探测。

2

缺失与超时检测

例如请求已经发出，但在预期窗口内没有收到响应、确认或回执。

3

上游打分，下游聚合

先对单条事件做语义 enrichment 和风险评分，再按实体或时间窗口聚合判断。

4

离线复盘与回放验证

用 batch、`wfl verify` 和 `wfgen` 在样本上复盘规则行为，并保持与线上语义一致。

快速开始

官网上的上手路径应该先帮助读者建立模型，再让读者跑通闭环。

1

理解三文件模型

先建立 `.wfs`、`.wfl`、`fusion.toml` 的职责边界，避免把业务规则和运行参数写在同一层。

2

跑通最小项目

用最小 schema、rule 和 config 启动 `wfusion` ，先理解实时接入或离线回放的执行闭环。

3

用 `wfl` 验证规则

优先 explain、lint、replay、test，把规则语义先定稳，再接入真实流量。

4

用 `wfgen` 建立闭环

通过 gen → send → verify 把规则正确性和运行时输出对拍起来。
